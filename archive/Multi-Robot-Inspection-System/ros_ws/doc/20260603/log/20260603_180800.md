# Task layer v0.2 completion update

## Context

The current work finished the `experiments/task_layer_v020` prototype as a usable v0.2 baseline for RoboInspect. The goal of this version is intentionally limited: single-robot semantic-route inspection in Gazebo/Nav2, simple photo evidence collection, report generation, and a unified operator GUI.

This version does not attempt visual recognition, LiDAR-based anomaly classification, precise docking, multi-robot assignment, or adaptive task planning.

## Package

New experimental package:

```text
experiments/task_layer_v020
```

The package remains under `experiments/` because it is still a task-layer prototype. Mature packages can later be moved to `src/`.

## Completed capabilities

### Bringup

`bringup_nav.launch.py` starts:

- Gazebo map world
- TurtleBot4 spawn
- localization
- Nav2
- RViz
- automatic AMCL initial pose

It uses the aligned map:

```text
/home/ryan/tb4_ws/maps/base_aligned_20260602.yaml
```

A v0.2 Nav2 config was added:

```text
config/nav2_inspection.yaml
```

This keeps TurtleBot4 defaults but tightens final yaw tolerance for more stable inspection-facing directions.

### Semantic world model

`config/world_model.yaml` contains:

- world metadata
- robot start at `charging_station`
- semantic area centers, sizes, bounds, and marker metadata

This file is the shared source for GUI navigation, inspection route selection, and scene placement.

### Navigation backend

`go_to_area` supports CLI semantic-area navigation through Nav2 `NavigateToPose`.

### Inspection backend

`inspect_area` supports:

- single area via `target:=...`
- route inspection via `route:=area_a,area_b,...`
- adaptive candidate inspection poses based on area bounds
- candidate attempt limit per area
- Nav fail evidence capture
- four-direction scan/camera capture after reaching an inspection pose
- continue-on-fail route execution
- return-home behavior to a safe standoff pose near charging station
- concise `report.yaml`
- full technical `details.yaml`

Important behavior choices:

- No LiDAR anomaly judgment in v0.2.
- No visual object classification in v0.2.
- Nav fail is recorded as execution status, not interpreted as obstacle detection.
- Reports are evidence/status records, not inspection intelligence.

### Scene/model spawning

`spawn_model` and the GUI scene tab can spawn built-in SDF models into the running Gazebo world. The model backend resolves model paths and supports repeated spawns with unique names.

Built-in test models include:

- `small_box_obstacle`
- `pallet_box_mobile`
- `shelf`
- pallet jack model

### Unified Task GUI

The old navigation GUI was upgraded and renamed to:

```text
task_gui
launch/task_gui.launch.py
```

The GUI now has three tabs:

```text
Navigate: semantic area Nav2 goals
Inspect: route builder and inspect_area launcher
Scene: model placement/spawn controls
```

The old `nav_gui` source and launch entry were removed. `scene_builder_gui` is still kept as a legacy focused scene-builder entry while the unified GUI matures.

## Report handling

Generated reports are stored under:

```text
experiments/task_layer_v020/reports
```

`reports/.gitignore` keeps generated inspection reports and images out of git while preserving the directory.

Recent manual test reports were pruned to the latest few runs during development.

## Validation status

Validated during interactive testing:

- Gazebo + localization + Nav2 launch works.
- Fixed semantic-area Nav works.
- Scene model spawning works.
- Custom inspection route can execute through the unified GUI/backend.
- Reports are generated in the expected folder format.
- Failure cases can produce Nav fail status and evidence screenshots.

Known behavior is still variable around dynamic obstacle placement and Nav2 local behavior. This is acceptable for v0.2 and should be addressed in later versions with clearer skill/fallback design rather than more ad-hoc logic.
