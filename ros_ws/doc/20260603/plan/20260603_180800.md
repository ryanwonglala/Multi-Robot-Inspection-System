# Next plan after task_layer_v020

## Current baseline

`task_layer_v020` is now the v0.2 baseline:

```text
single robot + semantic route + fixed inspection skill + evidence/report + unified GUI
```

This is enough for the current v0.2 milestone.

## Known limitations

- Inspection is still route-driven, not task-planner-driven.
- Candidate pose selection is heuristic and only uses semantic area bounds.
- Nav2 behavior around dynamic obstacles can be slow or inconsistent.
- Nav fail evidence is useful but not a true abnormality classifier.
- No YOLO or visual recognition is connected yet.
- No multi-robot namespace/assignment layer exists yet.
- Return-home uses Nav2 to a safe standoff pose, not precision docking.

## Recommended v0.3 direction

Do not keep adding one-off inspection heuristics into v0.2. The next version should introduce a cleaner task layer around the existing skills.

Suggested v0.3 goals:

1. Define explicit skill interfaces:
   - `NavigateToArea`
   - `InspectAreaRoute`
   - `CaptureEvidence`
   - `SpawnSceneObject`
   - `ReturnHome`
   - `ReportTask`
2. Add structured task input files for repeatable demos.
3. Add a task execution state machine rather than embedding all decisions in `inspection_runner.py`.
4. Add GUI support for loading/saving inspection route presets.
5. Add better execution feedback in GUI while an inspection is running.
6. Decide whether dynamic obstacle handling belongs to:
   - Nav2 tuning
   - alternate candidate strategy
   - explicit blocked-path skill
   - future perception module
7. Keep visual anomaly detection as a later module, not part of v0.2.

## Suggested immediate next step

Stabilize the unified `task_gui` experience:

- show live process output from `inspect_area`
- add a report list/open button
- add route preset save/load
- optionally remove legacy `scene_builder.launch.py` after GUI stability is confirmed

Then branch into v0.3 package rather than continuing to mutate v0.2 indefinitely.
