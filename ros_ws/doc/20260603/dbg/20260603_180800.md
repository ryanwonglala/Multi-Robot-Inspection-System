# task_layer_v020 debug notes

## Initial pose and map alignment

Earlier navigation failures were caused by initial pose orientation / coordinate alignment issues rather than the map being unusable. The aligned map and current start pose have been validated with Gazebo/RViz matching.

## LiDAR anomaly attempt removed

A previous v0.2 idea tried to infer obstacle abnormality from LiDAR point counts. Testing showed the result was unstable and not intuitive. That path was removed from v0.2.

Current policy:

```text
LiDAR scan summaries may be recorded in details.yaml, but report.yaml does not make abnormal/normal claims.
```

## Photo evidence

Successful inspection captures four directional images after reaching an inspection candidate pose.

Nav failure can capture immediate camera evidence as:

```text
nav_fail_attempt_XX.ppm
```

This is evidence of execution context, not a classified abnormality.

## Candidate poses

Candidate inspection poses were changed from fixed +/-0.5 m offsets to an area-bounds-based spread strategy. This helps large areas such as `north_hall` avoid all candidates clustering around the center.

Parameters:

```text
candidate_offset:=0.5
candidate_spread_ratio:=0.35
bounds_margin:=0.25
max_candidate_attempts_per_area:=2
```

## Timeout removal

A temporary per-goal timeout/cancel mechanism was tested but removed because it interrupted routes that were slow but still reachable. Nav2 action results are now allowed to complete according to Nav2 behavior.

## Return home

Returning directly to the charging marker/dock pose caused collision with the charging model. v0.2 now returns to a safe standoff pose in front of the charger:

```text
return_home_standoff_distance:=0.6
```

This is not precision docking.

## GUI issue fixed

The first `task_gui` launch failed because `use_sim_time` was declared twice. The manual declaration was removed from `TaskGuiNode`; launch still passes the parameter.

## Current known rough edges

- Dynamic obstacle behavior depends heavily on Nav2 local planner behavior.
- Some reachable routes may be slow around large obstacles.
- Candidate selection is still heuristic.
- The GUI currently starts inspection via subprocess and only reports completion/report path, not rich live progress.
