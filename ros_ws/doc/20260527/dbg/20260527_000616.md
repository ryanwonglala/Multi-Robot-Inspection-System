# Dbg 20260527_000616

## Symptom

Manual launch failed after the first local `map.launch.py` implementation.

Observed behavior:

- Gazebo starts and shows the custom map plus TurtleBot4 for a few seconds.
- Gazebo GUI exits soon after.
- RViz2 stays open.
- RViz Global Status reports frame `map` missing.
- Robot model is not loaded correctly in RViz.

## Log Findings

Recent ROS launch log showed:

```text
gazebo process has finished cleanly
spawner_joint_state_broadcaster could not contact controller_manager
spawner_diffdrive_controller could not contact controller_manager
```

Interpretation:

- `map` missing in RViz is likely a downstream symptom.
- SLAM did start, but with Gazebo/control unavailable it could not produce a valid map frame from live scan/tf data.
- Controller spawners failed because the Gazebo-side controller manager disappeared or became unreachable.

Older failed logs also showed malformed bridge topics like:

```text
/world//home/ryan/tb4_ws/src/sim/worlds/map/...
/world/map.sdf/...
```

This confirmed the bridge must receive the Gazebo world name only, not a file path or `.sdf` value.

## Fixes Applied

1. `src/sim/launch/map.launch.py` now explicitly passes `world` to official TurtleBot4 spawn/bridge launch.
2. `src/sim/worlds/map.sdf` and `src/sim/worlds/demo.sdf` no longer enable world-level `Sensors` with `ogre`.
3. Added `Contact` system plugin to match official TurtleBot4 worlds more closely.

## Reasoning

Official TurtleBot4 worlds include:

```xml
<plugin name="gz::sim::systems::Contact" filename="gz-sim-contact-system" />
```

Official TurtleBot4 worlds keep the world-level Sensors plugin commented out.

The custom world used:

```xml
<plugin name="gz::sim::systems::Sensors" filename="gz-sim-sensors-system">
  <render_engine>ogre</render_engine>
</plugin>
```

This may conflict with TurtleBot4's simulated camera/LiDAR stack or Gazebo Harmonic rendering assumptions.

## Verification Done

```bash
colcon build
source install/setup.bash
ros2 launch sim map.launch.py --show-args
```

Build and launch argument parsing passed.

## Follow-up 20260527_001708

Gazebo no longer exits after the world/launch fix.

New repeated warning:

```text
[diffdrive_controller]: Ignoring the received message (timestamp 21.0000000000) because it is older than the current time by 0.5000000000 seconds, which exceeds the allowed timeout (0.5000)
```

Relevant official config:

```yaml
cmd_vel_timeout: 0.5
use_stamped_vel: false
```

Interpretation:

- This is not the same failure as the earlier Gazebo exit.
- The warning means a velocity command reached the diffdrive controller at or just beyond the 0.5 s command timeout.
- If the robot still moves normally, this can be treated as noise during idle or low-rate command publishing.
- If the robot does not move, stutters, or ignores teleop, then this warning becomes important and the next target is command publish rate / sim time / cmd_vel bridge timing.
