# Dbg 20260527_013544

## User Report

Gazebo GUI manual save overwrote the original `map.sdf`. Opening the world directly in Gazebo showed the edited map, but launching through ROS still appeared to load the old map.

## Findings

The source world was changed:

```text
src/sim/worlds/map.sdf
size: 74699 bytes
mtime: 2026-05-27 01:31:27 +0800
```

The launch path uses the installed package share:

```text
/home/ryan/tb4_ws/install/sim/share/sim
```

The installed world path is:

```text
install/sim/share/sim/worlds/map.sdf
```

After rebuild, both files match:

```text
sha256: cc746e097d1dbb71fbc0174087f1d679ccb6eea659bad09fcb539074b6132e36
```

## Current Source Structure

Current source map has:

```text
models: 46
wall models: 44
extra wall suffix models:
- wall_passage_north_right_1
- wall_passage_north_left_1
- wall_prep_south_1
```

This confirms Gazebo GUI saved new/duplicated wall model entities into `src/sim/worlds/map.sdf`.

## Explanation

`gz sim src/sim/worlds/map.sdf` reads the source file directly.

`ros2 launch sim map.launch.py` resolves `sim` through the installed ROS package and reads:

```text
install/sim/share/sim/worlds/map.sdf
```

Therefore, after editing `src/sim/worlds/map.sdf`, run `colcon build` and re-source the workspace before launching through ROS.
