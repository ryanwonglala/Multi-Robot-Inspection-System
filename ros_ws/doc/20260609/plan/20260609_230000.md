## Known Limitations

### L1 — Map does not cover the southern facility (HIGH priority)

`maps/tb3_map.yaml` covers `Y[-1.45, 14.5]` only. The southern half of the
facility (`Y < -1.45`) is unmapped free space (white) in the costmap, causing
Nav2 to plan routes that pass through uncharted walls.

Areas unreachable for navigation (outside map boundary):
- entrance_lobby (center y=-5.7)
- main_corridor (center y=-2.6)
- mother_base (center y=-3.3)
- charging_station (center y=-3.825)
- storage_area (center y=-2.75)
- utility_area (center y=-2.9)
- narrow_passage (center y=-5.2)

### L2 — initial_pose mismatch (MEDIUM priority, blocked on L1)

`set_initial_pose_node.py` sends AMCL initial pose `x=0.0, y=0.0`.
Robot actually spawns at `x=-4.8, y=-3.825, yaw=-1.5708`.
This causes AMCL to localise incorrectly at startup; the operator must
manually set the pose in RViz.

Will be fixed after L1: update `set_initial_pose_node.py` default params
to match the spawn pose.

### L3 — east_hall / restricted_zone outside map east boundary (LOW priority)

`east_hall` extends to `x_max=9.37`, `restricted_zone` to `x_max=9.37`.
Current map east boundary is `x=3.45`. Both areas need a wider map to reach.
This is a secondary issue relative to the north–south coverage gap.

## Multi-Robot Namespace Design (Week-ahead task)

The current single-robot stack uses no namespaces. Adding Robot B requires:

1. **Simulation side** (`src/sim`):
   - Spawn Robot B under namespace `/robot_b` using `spawn_entity.py -robot_namespace robot_b`
   - Or launch two separate instances of `map.launch.py` with a namespace argument

2. **Nav2 side** (`src/task_layer`):
   - Each robot gets its own `nav2_bringup` instance under `/robot_a` and `/robot_b`
   - Topics: `/robot_b/cmd_vel`, `/robot_b/scan`, `/robot_b/amcl_pose`, etc.
   - Each robot needs its own `map_server` instance (same map) — or use `multirobot_map_merge`

3. **task_layer side** (`src/task_layer`):
   - `inspection_runner.py`: add `robot_ns` launch argument, pass to
     `NavigateToPose` action client namespace
   - `task_gui_node.py`: add robot selector dropdown (A / B)
   - Coordinate via `~/ros2_ws/multirobot_map_merge` already installed

**Reference:** `~/ros2_ws/` has `multirobot_map_merge` and `explore_lite`
ready to use.

**Key decision needed:** do both robots run independent Nav2 stacks (separate
costmaps, separate planners) or share a single global costmap through the
merge node? Recommendation: independent stacks with a merged display-only map
for the operator GUI. This decouples failure domains.
